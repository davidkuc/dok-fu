# Dok-Fu — System Instructions

## What is Dok-Fu

Dok-Fu is a documentation workflow system that connects source code and documentation through two-way pointers, a searchable index, and a controlled tag vocabulary. It is designed for incremental, AI-assisted documentation that stays in sync with a living codebase.

## Pillars

**Terseness** — minimal verbosity, maximal signal. Every layer has hard limits:
- Index entries: 1 sentence description maximum
- Module sections: ≤ 3 sentences, ≤ 5 bullet points per H2 section
- Inline code comments: 1 sentence maximum

**Progressive Disclosure** — avoid loading unnecessary context. Information is layered from coarsest to finest:
1. `docs/index.json` — quickest lookup; tags and 1-sentence descriptions only
2. Doc modules (`docs/**/*.md`) — medium detail; section-level summaries
3. Code comments — high detail; granular descriptions inside source files
4. Source code — full detail; last resort

**Deterministic Foundation** — scripts handle traversal, extraction, and validation. AI handles reasoning, writing, and orchestration. Never implement what a script can do reliably or if a script already handles a certain problem.

**AI Augmenting** — AI uses scripts as tools. Scripts produce structured output; AI interprets, writes, and decides.

## Format Rules

### Module boundaries

One doc module per **source directory**. A module documents a folder, not a single file. Each H2 section inside the module documents exactly one source file within that folder.

### Doc module frontmatter

```yaml
---
dokfu_id: <stable-slug>
code: <repo-relative-path-to-source-folder>
tags: [<tag1>, <tag2>]
description: <one sentence>
---
```

- `dokfu_id`: slugified folder path (e.g. `src-auth`); used for rename repair
- `code`: repo-relative path to the **source folder** this module covers; must resolve to an existing directory
- `tags`: values must exist in `config/tags.registry.json`
- `description`: one sentence; this is what appears in the index

### Doc module body
```
# <Component/Folder Name>

## Sections
- [filename.py](src/component/filename.py)
- [other.py](src/component/other.py)

## filename.py
path: src/component/filename.py
<content — max 3 sentences and 5 bullets>

## other.py
path: src/component/other.py
<content — max 3 sentences and 5 bullets>
```

The `## Sections` block lists the module's H2 headings — one per source file in the folder. Each H2 section header is the **filename**, and the first line of the section body is `path: <repo-relative path>` to that specific file. For root-level files, document them in `docs/root.md` with `code: .` in the frontmatter.

Section links use relative paths to the source file (e.g. `[login.py](src/auth/login.py)`).

### Code pointer
A single comment line placed near the top of **each source file** (after shebang/package, before imports). Points to the parent module (folder-level doc):

```
# dok-fu: docs/<mirrored-folder-path>.md
```

The canonical format shows `# dok-fu:` with a space after `#`. The comment token is determined by file extension using `comment_map` in `config/dok-fu.config.json`. All files within the same folder point to the same module doc.

### Index (`docs/index.json`)
Flat JSON array, never hand-edited. Regenerated by `dokfu index`. Each entry:

```json
{ "path": "docs/...", "tags": ["tag1"], "description": "One sentence." }
```

## Scripts Reference

| Command | Purpose |
|---|---|
| `dokfu index` | Rebuild `docs/index.json` from all module frontmatter |
| `dokfu index --check` | Exit non-zero if index is stale |
| `dokfu tags --list` | Print all tags in the registry with explanations |
| `dokfu tags --search <TAG>` | List doc paths that carry the given tag |
| `dokfu doctor` | Validate pointers, tags, and index freshness; report broken/orphaned |
| `dokfu doctor --fix-index` | Also rebuild index if stale |
| `dokfu doctor --fix-pointers` | Detect broken pointers; for rename candidates, update pointer comments in source files |
| `dokfu changes [--since <REF>]` | List source files changed since ref (git primary, manifest fallback) |
| `dokfu generate` | Regenerate `.github/` and `.claude/` from `base/` (idempotent) |
| `dokfu install [--target .]` | Full install into a target project |

## Skills Reference

| Skill | When to use |
|---|---|
| **Traverse** | Need to research the codebase without loading everything |
| **Enrich** | Source files lack doc pointers, modules, or index entries |
| **Update** | Source files changed and docs are stale |

### Choosing the right skill

- **Exploring unfamiliar code?** → Traverse
- **New file added / doc gaps detected by `dokfu doctor`?** → Enrich
- **`dokfu changes` returned a non-empty list?** → Update
- **Pointer broken / file renamed?** → Run `dokfu doctor` to detect rename candidates via `dokfu_id` matching; run `dokfu doctor --fix-pointers` to auto-repair pointer comments in source files; then run `dokfu index`

## Invariants

- Tags must always be from `config/tags.registry.json`. Never invent tags.
- `docs/index.json` is always regenerated by script, never edited by hand.
- A doc module and its source file must always point to each other (two-way).
- Terseness limits are hard constraints, not guidelines.
